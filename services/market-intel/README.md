# TradeIQ Market Intelligence Service

A single-file FastAPI microservice that powers TradeIQ's market data feed:

| Endpoint                     | What it does                                              |
|-------------------------------|------------------------------------------------------------|
| `GET /healthz`                 | Liveness/readiness probe                                   |
| `GET /market/snapshot`          | NGX daily market data (ASI, volume, market cap, breadth)   |
| `GET /market/news`              | Latest NGX-related headlines                               |
| `GET /market/news/explained`    | Same headlines, rewritten in plain English via Groq/Llama  |

## Quick start

```bash
cp .env.example .env
# edit .env and fill in your real NGX / NewsAPI / Groq keys

pip install -r requirements.txt
uvicorn main:app --reload
```

Then visit `http://localhost:8000/docs` for interactive Swagger docs (auto-generated by FastAPI).

## Docker

```bash
docker build -t tradeiq-service .
docker run -p 8000:8000 --env-file .env tradeiq-service
```

## Why it's built this way

**One file, three upstreams.** `main.py` is intentionally kept as a single
readable module — config, caching, the three upstream integrations, and the
routes that expose them. At this size, splitting into multiple files would
add navigation overhead without a real maintainability payoff. If you later
add a 4th or 5th data source, that's the natural point to split into
`upstreams/ngx.py`, `upstreams/news.py`, etc.

**No hardcoded secrets.** Every API key is read from the environment via
`pydantic-settings`, which also validates at startup — if a required key is
missing, the service fails fast with a clear error instead of failing later
mid-request.

**Async all the way through.** A single shared `httpx.AsyncClient` is reused
across requests (connection pooling), and the Groq SDK calls (which are
synchronous) are pushed to a thread pool via `asyncio.to_thread` so they
never block the event loop. This is what lets one process handle many
concurrent requests instead of queuing them.

**Caching in front of every upstream.** NGX/NewsAPI/Groq are all rate-limited
and/or billed per call. Each upstream has its own short-lived in-memory TTL
cache (`CACHE_TTL_SECONDS`, default 60s) so a burst of requests from your app
collapses into one upstream call. The Groq explanation cache is held 5x
longer by default, since LLM calls are the most expensive part of the
pipeline.

**Errors never leak as stack traces.** Every upstream call is wrapped and
turned into a clean `UpstreamError` with an HTTP status and message. A
top-level middleware also catches anything unexpected, logs it with a
request ID, and returns a generic 500 — so a client never sees a raw
traceback, but you can always find the matching detailed log line by request
ID.

**Structured logs with request IDs.** Every request gets a short ID that's
attached to every log line for that request (and returned in the
`X-Request-ID` response header), so you can trace one request through logs
even under concurrent load.

## Scaling this in production

- Run multiple workers/replicas (`--workers N` or multiple containers behind
  a load balancer) — the service holds no per-worker state that isn't safe to
  duplicate; the caches are just a performance optimization, not a source of
  truth.
- If you outgrow in-memory caching (e.g. multiple replicas duplicating
  upstream calls), swap the `TTLCache` instances for Redis with the same TTL
  semantics — the `cached()` decorator's interface would stay the same.
- Add an API gateway / reverse proxy (nginx, Caddy, or your cloud LB) in
  front for TLS termination and rate limiting from your own clients.
